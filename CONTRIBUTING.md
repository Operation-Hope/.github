## Code of Conduct

This project and everyone participating in it is governed by our Code of Conduct. By participating, you are expected to uphold this code. Please report unacceptable behavior to the project maintainers.

## Getting Started

1. **Fork the repository** on GitHub
2. **Clone your fork** locally
3. **Create a branch** for your changes:
   ```bash
   git checkout -b your-alias/your-feature-name
   ```
4. **Set up your development environment** by following the instructions in the README.md

## How to Contribute

### Reporting Bugs

Before creating bug reports, please check the existing issues to avoid duplicates. When you create a bug report, include as many details as possible:

- **Use a clear and descriptive title**  
- **Describe the exact steps to reproduce the problem**  
- **Provide specific examples** to demonstrate the steps  
- **Describe the behavior you observed** and what you expected to see  
- **Include screenshots or animated GIFs** if applicable  
- **Note your environment details** (OS, browser, version, etc.)  

### Suggesting Enhancements

Enhancement suggestions are tracked as GitHub issues. When creating an enhancement suggestion, include:

- **Use a clear and descriptive title**  
- **Provide a detailed description** of the suggested enhancement  
- **Explain why this enhancement would be useful** to most users  
- **List examples** of how the feature would be used  
- **Note if similar features exist** in other projects  

### Pull Request Process

1. **Ensure your code follows** the project's style guidelines  
2. **Update documentation** as needed, including the README.md  
3. **Add tests** for new functionality when applicable  
4. **Ensure all tests pass** before submitting  
5. **Update the CHANGELOG** if one exists  
6. **Reference any related issues** in your PR description (e.g., "Fixes #123")  
7. **Request review** from maintainers  
8. **Be responsive to feedback** and make requested changes promptly  

#### PR Title Format

Use clear, descriptive titles following this convention:
- `feat: Add new feature`
- `fix: Resolve issue with...`
- `docs: Update documentation for...`
- `style: Format code in...`
- `refactor: Restructure...`
- `test: Add tests for...`
- `chore: Update dependencies`

## Style Guidelines

### Git Commit Messages

- Use the present tense ("Add feature" not "Added feature")
- Use the imperative mood ("Move cursor to..." not "Moves cursor to...")
- Limit the first line to 72 characters or less
- Reference issues and pull requests after the first line
- Consider starting the commit message with an applicable emoji for better readability

### Code Style

- Follow the existing code style in the project
- Write clear, self-documenting code with meaningful variable names
- Add comments for complex logic
- Keep functions focused and modular
- Ensure code is properly formatted before committing

### Documentation Style

- Use Markdown for documentation
- Keep line length to 80-100 characters when possible
- Use clear, concise language
- Include code examples where helpful
- Keep documentation up to date with code changes

## Development Workflow

1. **Pick an issue** to work on or create a new one
2. **Comment on the issue** to let others know you're working on it
3. **Create a branch** from the main branch
4. **Make your changes** following the style guidelines
5. **Test your changes** thoroughly
6. **Commit your changes** with clear commit messages
7. **Push to your fork** and submit a pull request
8. **Participate in the review process**

## Testing

When applicable:

- Write tests for new features and bug fixes
- Ensure all existing tests pass
- Run the test suite locally before submitting a PR
- Aim for good test coverage
